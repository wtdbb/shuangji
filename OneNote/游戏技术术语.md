## 引言

在计算机中，内存是非常重要的，空间和计算机的算力都是有限的，占用更多的空间或者从头到尾把数据全部找一遍非常浪费时间，你也不希望刷个qq卡个半死对吧，那算法，架构等一系列方法都是为了减少占用内存，所以有百分之60的方法都是为了减少内存或者减少代码运行时间，这种思维我们叫做时间复杂度或者空间复杂度，所以大家看的时候一定要带着怎么减少代码运行时间和内存空间占用去理解。

另外单独解释下算法，这个是特别难的部分，因为需要极强的数学思维能力，一个好的算法甚至可以获得诺贝尔奖--------那什么是算法呢：用个最简单的例子：从0-100，有个炸弹数字，用最少得时间找出这个数字，只能说大了或者小了，我们可以猜50，然后发现大了，那就猜0-50；这就叫二分查找算法，而不是从头到尾数一遍

所以经过上面一段话也相信大家知道了内存，代码运行时间的重要性，那就请大家看接下来的内容吧~

## M0:常见岗位介绍

1. 客户端

> 一句话核心：让玩家能看到、能操作的那部分程序。

**日常工作：**

实现游戏界面（按钮、血条、背包）

处理玩家输入（点击、滑动、按键）

控制角色移动、释放技能、播放动画

把服务器发来的数据显示在屏幕上

**通俗比喻：餐厅里端菜给客人、摆盘、收拾桌子的服务员。客人看到和接触到的都是客户端负责。**

**常见技术：Unity（C#）、UE（C++）、Cocos（TS/JS）**

2. 服务器

> 一句话核心：玩家看不见、但所有重要数据都在它那里的后台程序。

**日常工作：**

接收所有玩家的操作（移动、攻击、聊天）

判定逻辑（打中了没、扣多少血）

存储数据（等级、装备、充值记录）

把结果同步给所有相关玩家

**通俗比喻：餐厅的后厨 + 仓库 + 收银台。客人看不到后厨，但菜从那里出，账从那里算。**

**常见技术：Go、C++、Java、Erlang**

3. 测试

> 一句话核心：在玩家玩到之前，把游戏里的坑踩一遍。

**日常工作：**

按测试用例反复操作（点遍所有按钮、走遍所有地图）

发现bug后写清楚复现步骤，提交给程序修

验证bug是否真的修好了

有些测试会写自动化脚本，让机器帮忙跑重复工作

**通俗比喻：餐厅的试吃员和质检员。菜上给客人之前，他们先尝咸淡、检查有没有头发。**

**常见分类：功能测试（只测好不好用）、性能测试（测卡不卡）、自动化测试（写代码测）**

4. 技术美术（TA，Technical Artist）

> 一句话核心：美术和程序之间的翻译官+胶水。

**日常工作：**

帮美术实现各种视觉效果（但不一定自己画）

优化资源，让游戏跑得更流畅

给美术写工具，比如批量处理贴图、自动生成LOD

告诉程序“美术想要的效果技术上能不能做、怎么做”

**通俗比喻：餐厅里懂做菜又懂财务的店长。主厨（美术）说想加一道菜，他能判断后厨能不能做、成本够不够，还能帮忙优化流程。**

**常见方向：渲染TA（搞光影特效）、工具TA（写插件）、材质TA（调shader）**

5. 引擎开发

> 一句话核心：做引擎本身的人，而不是用引擎做游戏的人。

**日常工作：**

维护和优化引擎底层（渲染管线、物理系统、资源加载）

给客户端程序员提供更好用的功能接口

适配不同平台（PS5、Switch、手机）

解决引擎本身的bug或性能瓶颈

**通俗比喻：造炉灶和锅的人，而不是用锅炒菜的厨师。UE、Unity的开发人员就是引擎开发。**

**注意：大部分游戏公司不需要这个岗位，只有引擎公司（Epic、Unity）或大厂自研引擎（如腾讯、米哈游）才设。**

> 一句话总表（便于对比）

| 岗位 | 一句话 | 产出物 |
| --- | --- | --- |
| 客户端 | 玩家看到和操作的 | 游戏界面、逻辑、画面表现 |
| 服务器 | 玩家看不到的后台 | 数据存储、逻辑判定、消息同步 |
| 测试 | 提前踩坑的人 | bug报告、测试用例 |
| 技术美术 | 美术和程序的桥梁 | 效果实现方案、优化工具 |
| 引擎开发 | 做工具的人 | 引擎底层功能、开发接口 |

另外因为服务器和客户端有多个语言和引擎，并不互通，比如做unity的不能去ue，所以详细再介绍一下客户端部分共同点（联系）

都是用来开发游戏画面的工具

都要处理：渲染、物理、输入、UI、音频

都能发布到多个平台（PC/手机/网页）

### 区别一览表

| 引擎 | 核心语言 | 擅长领域 | 代表作 | 上手难度 |
| --- | --- | --- | --- | --- |
| Unity | C# | 3D/2D手游、独立游戏 | 原神、王者荣耀 | 低 |
| UE | C++ | 3A级PC/主机游戏、高画质手游 | 黑神话悟空、和平精英 | 高 |
| Cocos | TS/JS | 2D小游戏、卡牌、H5 | 开心消消乐 | 很低 |

### 一句话选型

要做《原神》级别的手游 → Unity

要做《黑神话》级别的PC游戏 → UE

要做微信小游戏或2D卡牌 → Cocos

### 服务器部分

### 共同点（联系）

都能写服务器后端

都要处理：网络通信、数据存储、玩家状态同步

都能配合数据库（如MySQL/Redis）

### 区别一览表

| 语言 | 性能 | 并发模型 | 开发效率 | 典型游戏 | 一句话特点 |
| --- | --- | --- | --- | --- | --- |
| C++ | 极高 | 手动管理 | 低 | 腾讯系大型网游 | 最强性能，但写着累 |
| Java | 高 | 多线程 | 中 | 很多MMO（早期） | 稳定，吃内存 |
| Go | 高 | 协程（天生高并发） | 高 | 棋牌、中轻度手游 | 现在最火的后端选择 |
| Erlang | 高 | Actor模型 | 低 | 微信游戏、实时聊天 | 容错极强，但小众 |
| Python | 低 | 全局锁限制 | 很高 | 游戏后端原型、工具链 | 开发快，性能是瓶颈 |

### 近几年的趋势

新项目大多偏爱 Go（够快 + 写起来简单）

追求极致性能（如FPS、MOBA）用 C++

老项目或大公司稳定业务用 Java

Erlang 和 Python 逐渐边缘化，除非有特殊需求

### 基础部分讲完了，现在开始正题，接下来的内容是帮助大家看懂简历

## 前置篇：先把一份程序简历拆成三个维度

> 方向决定他做什么，阶段决定他经历过什么，技术线决定他的能力深度。

> 推荐阅读顺序第一步：先看候选人属于客户端、服务端、TA、测试还是引擎开发。第二步：再给他打三个标签——业务方向、项目阶段、技术线。第三步：用“技术含金量分级”判断深度，不要只看关键词数量。第四步：遇到陌生术语，再去后面的词典章节查。第五步：最后用文末的实力判断表做交叉验证。

### 一、业务方向：他具体在游戏里做哪一块

- ❶ 核心玩法层
- ▎ 玩法（Gameplay）：把策划的玩法案变成代码。副本逻辑、PvP 模式、回合制规则等都在这。简历里看「完整玩法模块从 0-1」是关键信号。
- ▎ 战斗（Combat）：玩家挥刀、技能命中、伤害计算、霸体浮空、buff/debuff —— 所有打架相关的底层。米哈游、鹰角这类动作游戏的核心命脉。
- ▎ 3C（Camera / Character / Control）：相机、角色、操作手感。比如《原神》的镜头跟随、《黑神话》的闪避手感都是 3C 在调。动作游戏门槛最高的方向之一。
- ▎ 关卡（Level）：关卡内的逻辑实现 —— 触发器、机关、刷怪点、流程推进。和「关卡策划」对接最紧密。
- ▎ AI：怪物 AI、NPC 行为树、寻路（NavMesh）、感知系统。射击 / 动作游戏里很重要。
- ❷ 系统层
- ▎ 系统（System）：通用模块 —— 背包、任务、商店、邮件、社交、公会、活动框架……一切「非战斗」的功能。新手最容易切入。
- ▎ UI：界面框架、HUD、UI 动效。看似简单，但大型项目里 UI 性能、多分辨率适配是硬骨头。
- ▎ 网络 / 同步（Network）：多人同步、状态同步 vs 帧同步、延迟补偿、回滚。MOBA、射击、MMO 必备。
- ▎ 物理（Physics）：物理仿真、碰撞、布料、流体。《荒野大镖客》《GTA》那种沉浸感靠这个。
- ▎ 动画（Animation）：动画状态机、Motion Matching、IK、动画蓝图。和 3C / 战斗紧密耦合。
- ▎ 载具（Vehicle）：单独拎出来是因为载具系统物理 + 操控 + 网络同步都要专门处理。米哈游 Varsapura 项目就专门招载具客户端。
- ▎ 表现 / 特效（VFX / 演出）：技能特效、过场演出、序列帧、Sequencer / Timeline。介于程序和 TA 之间。
### 二、项目阶段：他经历过项目的哪一段

| 阶段 | 通俗说法 | 含义 |
| --- | --- | --- |
| 立项 | 概念阶段 | 老板拍板要做什么品类、什么风格，可能还没几个人 |
| 预研 | 技术 / 玩法验证 | 「这事能不能做」—— 比如能不能在手机上跑出主机画质，玩法 fun 不 fun。米哈游一堆「预研项目」就是这阶段，人少但要顶尖 |
| Demo / 原型 | 可玩 demo | 玩法跑通了，可以给老板 / 投资人看，但只有核心循环 |
| 垂直切片Vertical Slice | 「样板间」 | 用 1 个完整关卡把最终品质展示出来，所有美术 / 玩法都打磨到上线水准。立项过了之后给上级或发行方汇报用 |
| Alpha | 内部可玩版本 | 功能基本完整但没 polish，bug 一堆 |
| 一测 / CBT1 | 技术测试 / 删档测试 | 小范围（几千～几万人）放出去测压力、测玩法。鹰角、米哈游招人时常说「对标已上线项目」，但内部还在一测 |
| 二测 / CBT2 | 不删档 / 付费测试 | 验证商业化 |
| 三测 / 终测 | 上线前最后压测 | — |
| 公测 / OBT / 上线 | 正式开服 | 全面开放、商业化全开 |
| 运营期 / Live Ops | 长期运营 | 版本迭代、活动、商业化优化。明日之后、王者荣耀、原神长期都在这阶段。大头岗位都在这里 |
| 维护期 / 末期 | 缩编维护 | 减少更新，主要修 bug 保运营 |

▶ 对做单的实际意义

- ▎ 预研 / Demo 期：人少、要求高，看 0-1 经验和成功项目背景；钱多但风险高（可能立不了项）。例子：米哈游所有「预研」、鹰角「在研项目（搜打撤）」
- ▎ 垂直切片 / Alpha 期：补人最猛，要「快速上手做事」的资深执行向，不要太贵的管理岗
- ▎ 一测 / 二测期：开始招优化、安全、运维相关。客户端会大量招「性能优化向」
- ▎ 运营期：补「商业化、活动、外观」相关，以及「流失客户端开发」补编。要求「完整生命周期经验」「带过版本」的就是运营期需求。波克城市、网易明日之后都是典型
### 三、技术线：同样是客户端，技术深度可能完全不同

- ❶ 对外 / 接口类
- ▎ SDK 向：接入第三方 SDK —— 支付（微信 / 支付宝 / AppStore）、登录（微信 / QQ / Google / Apple）、推送、广告（穿山甲 / AdMob）、统计（GA / AppsFlyer）、客服等。看似杂活，其实是上线前必经路径，海外发行项目尤其重。Garena、字节出海项目这类需求多
- ▎ 平台 / 渠道：对接 App Store、Google Play、Steam、Epic、TapTap、各家手机厂商应用商店的认证、合规、分发。和 SDK 向重合
- ❷ 工具 / 基建类
- ▎ 工具向（Tools / Editor）：在引擎里写编辑器扩展，给策划 / 美术做关卡编辑器、技能编辑器、剧情编辑器、配置表工具、自动测试工具。提效核心岗，做得好一个工具开发顶 3 个执行
- ▎ 构建 / 管线（Build / CI-CD / DevOps）：乐元素白银之城那个「客户端-构建向」就是这个。负责自动化打包、Jenkins 流水线、出包出母包、版本号管理。派络特「中台 CI/CD」也是这方向。市面少有人专做、稀缺
- ▎ 基建（Infrastructure / Framework）：客户端底层框架 —— 模块化架构、生命周期、消息总线、资源管理。新项目立项时招的「基建主程」就这个。乐元素「客户端-基建性能优化」就是
- ▎ 热更新（Hot Update）：Lua 热更、AssetBundle 资源热更、增量更新。手游标配，主机 / Steam 不需要
- ❸ 引擎 / 底层类
- ▎ 引擎开发（Engine）：改 UE / Unity 源码、定制渲染管线、自研引擎。和「客户端」是平行岗位，技术深度更高，C++ 硬功夫
- ▎ 渲染 / 图形（Rendering / Graphics）：Shader、PBR、光照、阴影、后处理。和 TA 接壤但更偏底层
- ▎ 性能优化（Performance）：CPU / GPU / 内存 / 包体 / 启动速度 / 发热。运营期项目硬需求
- ❹ 安全 / 数据类
- ▎ 反外挂 / 安全（Anti-cheat / Security）：射击、MOBA 必备，对抗外挂、加密通信、检测篡改。腾讯网易这类大厂常设独立组
- ▎ 测试开发（Test Dev / QA Tools）：写自动化测试工具、性能监控、AI 玩家。网易招的「测试开发」就是。介于客户端和 QA 之间
❺ 跨界类

- ▎ TA / 技术美术：程序-美术桥梁。本质是程序，但服务于美术效果。乐元素一堆 TA 岗位就是
- ▎ 客户端 AI：把 LLM / Agent 接到游戏里。腾讯「光子技术发展部」那种岗位、网易天下 3 那个「游戏 AI 技术策划」是它的策划侧

怎样看“UI / 系统很多”

UI 和系统并不是没价值。问题在于：如果简历大篇幅都是“做页面、接需求、背包、商店、活动、邮件、任务”，但没有框架设计、性能优化、复杂交互、多人协作或从 0-1 证据，通常只能说明业务执行经验比较多。

反过来，如果写的是 UI 框架、MVVM、跨平台输入、动态列表、DrawCall、GC、资源释放、编辑器工具，或者系统模块的通用框架、数据驱动和长期维护，这类 UI / 系统候选人仍然可能很强。

怎样看“Performance 很多”

如果简历能成体系地写出 CPU / GPU / 内存 / 包体 / 启动速度 / 加载 / 发热，并且说明用了什么工具、找到了什么瓶颈、改完提升了多少，通常是优质信号。因为这类工作需要看懂整个项目，而不是只把一个需求做出来。

> 最重要的判断规则名词只说明“接触过”；动词说明“做到了哪一步”；数据说明“结果是否可信”。“使用 GAS” < “负责 GAS 技能模块” < “重构 GAS 预测与效果结算并解决线上同步问题”。

## 第一章 编程语言与内存基础

> C++、C#、Lua、Go，以及简历里常见的内存、并发和热更新词。

#### 1.1 C++ 与内存

#### 1.1 C++

> 例子：编程语言像不同工具：Python 适合做人工智能，Java 常用来做公司的大型系统，C# 常和 Unity 搭配。那C++ 的特殊本事是能自己精细管理内存，什么时候申请、什么时候释放都能管，所以跑得快、控制力强，但如果忘记释放或用错地址也更容易崩。UE、引擎底层和性能优化里经常见到它。

> 一句话：C++就是“控制权很大、性能很高，但也更容易写错”的语言。ps：属于难度较高的语言

#### 1.2 指针、引用与空值

> 例子：指针像一张写着门牌号的纸条，顺着地址能找到那份数据；引用像给同一个人起了外号，喊“Corvin”和喊“胡文聪”找到的是同一个人。空指针则像纸条上根本没写门牌号，拿着它去敲门当然找不到人。

> 一句话：指针存地址，引用是同一个对象的另一个名字，空指针表示“现在谁也没指”。

#### 1.3 对象生命周期与悬空问题

> 例子：公司给你一把储物柜钥匙。柜子从分配给你到退租，这段时间就是“生命周期”。柜子已经退租、甚至被分给别人了，但你手里还留着旧钥匙和旧门牌，这就是“悬空指针或悬空引用”。你再拿旧钥匙去开，轻则打不开，重则碰到别人的东西，程序里就可能崩溃或把数据改乱。

> 一句话：对象已经没了，指向它的旧地址还在，继续使用就是悬空问题。

#### 1.4 RAII

> 例子：进酒店房间时电卡一插，灯就亮；退房把电卡一拔，灯和空调自动关闭。RAII 就是把“申请资源”和“自动归还资源”绑在一起，对象离开作用范围时自动释放内存、文件或锁，不靠程序员临走前再想起来。

> 一句话：RAII就是“进门自动开、出门自动关”，让资源不容易忘记释放。

#### 1.5 unique_ptr、shared_ptr 与 weak_ptr

> 例子：unique_ptr 像全世界只有一把的保险箱钥匙，不能复制，只能把钥匙交给别人；shared_ptr 像一个微信群，群里还有人时资源就留着，最后一个人退群时资源才释放；weak_ptr 像潜水观察员，能看看群还在不在，但不算群人数，也不会拦着群解散。

> 一句话：unique_ptr 独占，shared_ptr 共享并计数，weak_ptr 只观察、不延长资源寿命。

#### 1.6 移动语义

> 例子：搬仓库有两种办法：一种是把所有货重新复制一份运过去；另一种是直接把仓库钥匙交给对方。移动语义就是后者，原来的人不再拥有货物，新的人直接接手，省下大量复制时间。

> 一句话：移动语义就是“资源不重抄，直接把所有权交过去”。

#### 1.7 虚函数、vtable 与运行时多态

> 例子：你手里只有一个写着“播放”的通用遥控器。对着电视按，电视播放画面；对着音响按，音响播放声音。同样的遥控却可以命令不同的家具，这就是运行时多态；虚函数就是掌握实现这个运行时多态的方法，有了他就可以实现

vtable是指虚函数表：程序运行时会先查一张类似通讯录的 vtable，上面写着所有的虚函数，然后确认眼前到底是什么对象，再调用它自己的版本，实现快速查找，不用从头找到尾

> 一句话：同一个函数名，面对不同对象会执行不同实现，这就是运行时多态。

#### 1.8 STL 容器

> 例子：STL 像 C++ 自带的一套收纳工具。vector 像一排连续货架，按“第几个”取东西很快，但在中间插一个要把后面都挪开；list 像用绳子串起来的箱子，中间接一个很方便，但找第 100 个得从头一路数。项目会按使用方式选容器，不用每次自己重写数据结构。

> 一句话：STL 是现成的数据容器和算法，重点不是背名字，而是知道不同容器适合不同用法。

#### 1.9 多线程、竞态与锁

> 例子：多个厨师一起做饭本来更快，但两个人同时改同一张订单，一个写“不要辣”，另一个写“特辣”，最后结果就乱了，这叫竞态。锁像规定同一时间只能一个人拿订单本，能避免冲突，但锁太多又会让大家排队。

> 一句话：多线程是多人同时干活，难点是别同时乱改同一份数据。

#### 1.10 对象池与内存池

> 例子：一次性纸杯每用一个都去工厂现造，用完就扔，既慢又浪费。对象池是提前备一批杯子，用完洗好放回去，下次继续拿；游戏里的子弹、特效、怪物和网络消息经常这样反复利用。

> 一句话：对象池就是“常用对象循环利用”，减少频繁创建、销毁和卡顿。

#### 1.2 C#、Lua 与 Go

#### 1.11 C# 与 GC

> 例子：C# 像住在有物业的公寓，平时不用自己搬走每袋垃圾，GC 会自动把没人再用的对象收走。好处是省心，坏处是保洁集中大扫除时可能停一下，Unity 项目如果每帧不停制造小垃圾，就容易出现 GC 卡顿。

> 一句话：C# 写起来省事，GC 会自动回收内存，但回收时也可能让游戏突然卡一下。

#### 1.12 协程、Task 与 async/await

> 例子：点外卖后不用一直站在门口等，可以先回去工作，外卖到了再继续处理。协程和 async/await 都是在“等待”时先让出时间，不过协程常跟着游戏帧往后接着跑，Task 更像一份可以异步完成的工作。

> 一句话：它们都是“先去做别的，等结果到了再接着干”，不等于一定新开了线程。

#### 1.13 Lua、xLua 与热更新

> 例子：游戏的大机器用 C++ 或 C# 做好后，把经常改的活动、界面和技能规则写在一张可替换的 Lua 说明书里。xLua 像翻译，让 C# 和 Lua 能互相调用。这样改活动时不用把整个游戏重新上架，只下发新脚本就行。

> 一句话：Lua 常被当成可替换的业务脚本，方便手游热更新。

#### 1.14 Go、goroutine 与 channel

> 例子：传统做法像每来一桌客人就雇一个全职服务员，成本很高；goroutine 像把很多小订单交给一组服务员统一调度，所以能同时照顾大量连接。channel 像厨房和前台之间的传菜口，用来安全传递消息。

> 一句话：Go 适合高并发服务端，goroutine 是轻量任务，channel 用来传消息。

## 第二章 游戏数学与空间计算

> 向量、矩阵、四元数、插值和碰撞体，只保留项目里常见的说法。

#### 2.1 位置、方向与旋转

#### 2.1 向量、点乘与叉乘

> 例子：向量像一根带方向的箭头。点乘可以快速判断目标是不是在角色前面：数值越大，方向越接近；叉乘像右手比出的垂直方向，常用来判断目标在左边还是右边，或求一个面的朝向。

> 一句话：向量管方向和距离，点乘看“朝不朝向”，叉乘看“左还是右、面朝哪”。

#### 2.2 4×4 矩阵与坐标变换

> 例子：角色站在车里时，先要知道他离车门多远，再把整辆车放到世界里的位置。矩阵像一张完整的搬运说明，能把缩放、旋转和平移一起算进去；而且先转再走和先走再转，结果可能完全不同。

> 一句话：矩阵就是把物体从一个坐标系搬到另一个坐标系的整套规则。

#### 2.3 欧拉角与四元数

> 例子：欧拉角像依次拧三个旋钮，直观但某些角度下两个旋钮会重合，出现“万向节死锁”。四元数像换了一套不直观但更稳定的旋转记录方式，角色转身、摄像机和网络旋转同步里经常用。

> 一句话：欧拉角好理解，四元数更适合稳定地保存和插值三维旋转。

#### 2.4 Lerp、Slerp 与 Tween

> 例子：电梯从 1 楼到 10 楼，Lerp 像按直线和时间比例往上走；Slerp 像沿球面平滑转动镜头；Tween 则像现成的动画遥控器，告诉它“从 A 到 B、用两秒、先慢后快”就能自动播放。

> 一句话：Lerp 管直线过渡，Slerp 管旋转过渡，Tween 把各种数值过渡包装成工具。

#### 2.5 AABB、OBB 与碰撞盒

> 例子：搬家时先拿一个大纸箱罩住家具，两个纸箱没碰到，里面的家具肯定没碰到；AABB 是始终和世界坐标对齐的纸箱，计算快。OBB 会跟着物体旋转，更贴合，但判断更麻烦。

> 一句话：AABB 便宜但粗，OBB 更准但更贵，常用来做碰撞的第一层筛选。

## 第三章 Unreal Engine 客户端

> Gameplay Framework、Lyra、GAS、CMC、动画、网络、DS、UI 与打包。

#### 3.1 UE 对象与项目框架

#### 3.1 UObject、Actor 与 Component

> 例子：普通 C++ 对象像没有登记的临时住户，UE 编辑器不一定认识它；UObject 像办了引擎身份证，能被反射、序列化和 GC 管理。Actor 像一整辆车，Component 则是发动机、轮胎和音响等可拆模块。

> 一句话：UObject 是 UE 的基础对象体系，Actor 是场景里的对象，Component 是挂在对象上的功能零件。

#### 3.2 Gameplay Framework

> 例子：一场球赛里，GameMode 是裁判，GameState 是大家都能看的记分牌，PlayerController 是玩家手里的遥控器，PlayerState 是选手信息牌，Pawn/Character 是被控制的角色，GameInstance 是换地图后还留着的总控室。

> 一句话：Gameplay Framework 是 UE 对“规则、玩家、角色和全局状态”的标准分工。

#### 3.3 Lyra 与 Game Features

> 例子：Lyra 像 Epic 提供的一套样板房，不是直接换皮就上线，而是展示大型 UE 项目怎么接 GAS、输入、UI 和多人网络。Game Features 像可插拔活动包，需要某种玩法时装上，不需要时关掉。

> 一句话：Lyra 是 UE 官方大型项目示例，Game Features 用来把玩法做成可插拔模块。

#### 3.4 CommonUI 与 Enhanced Input

> 例子：同一个游戏既要支持键盘、手柄又要支持触屏。Enhanced Input 把“跳跃”定义成动作，再分别绑定空格键和手柄 A；CommonUI 像菜单总管，处理多个页面叠在一起时谁接收输入、返回键退到哪里。

> 一句话：Enhanced Input 管输入动作，CommonUI 管跨平台菜单和焦点。

#### 3.2 GAS 能力系统

#### 3.5 GAS

> 例子：如果每个角色的技能都自己写一堆 if-else，项目越大越乱。GAS 像统一的技能工厂：技能怎么启动、消耗什么、加什么状态、能不能被打断、怎样联网，都走同一套流程。

> 一句话：GAS 是 UE 的完整技能与状态框架，常见于战斗、系统和网络岗位。

#### 3.6 ASC 与 AttributeSet

> 例子：ASC 像角色身上的技能总管，负责登记有哪些技能和效果；AttributeSet 像角色的数值账本，里面放生命、攻击、防御、能量等。技能要扣蓝或加血，最终都要改这本账。

> 一句话：ASC 管技能和效果，AttributeSet 管角色属性。

#### 3.7 GameplayEffect、MMC 与 Execution

> 例子：GameplayEffect 像一张药品说明：加多少血、持续多久、能不能叠加。MMC 像按等级和攻击力计算药量；Execution 像结算窗口，一次把攻击、防御、暴击、减伤等复杂公式算完。

> 一句话：Effect 描述效果，MMC 算一个数值，Execution 负责一整次复杂结算。

#### 3.8 GameplayTag

> 例子：给角色贴“眩晕、空中、无敌、正在蓄力”等统一标签。技能释放前先看标签：有“眩晕”就不能放技能，有“无敌”就不扣血。标签还能按层级查询，例如“状态.控制.眩晕”。

> 一句话：GameplayTag 是全项目统一的状态标签，GAS 会用它判断技能能不能放、效果能不能生效。

#### 3.9 GAS 预测与服务器校正

> 例子：玩家按技能后如果先等服务器回信，动作会显得很迟。预测就是客户端先把动作、扣蓝和特效演出来；服务器随后对账，同意就继续，不同意就撤销或修正。

> 一句话：GAS 预测让技能先有反馈，再由服务器确认最终结果。

#### 3.3 移动、动画与网络

#### 3.10 CMC（Character Movement Component）

> 例子：玩家本地先让角色跑起来，服务器同时记一份真正的移动账本。两边对不上时，CMC 会按规则把角色校正回来，还要尽量用平滑方式处理，避免玩家明显感觉被“拽回去”。

> 一句话：CMC 是 UE 角色移动、网络预测和位置校正的核心组件。

#### 3.11 Root Motion

> 例子：普通移动像程序拉着角色模型往前走，脚步动画只是表演；Root Motion 像动画本身真的把角色带过去，常用于冲刺、翻滚、处决和位移技能。

> 一句话：Root Motion 就是“位移写在动画里”，动作和距离更容易对齐。

#### 3.12 AnimBP、状态机、Blend Space、Montage 与 IK

> 例子：AnimBP 像动画总导演；状态机决定待机、跑步还是受击；Blend Space 按速度和方向混合走跑动画；Montage 像临时插播技能动作；IK 则像最后伸手调位置，让脚踩地、手握住武器。

> 一句话：这些词一起出现，通常表示候选人在做角色动画逻辑或战斗表现。

#### 3.13 Replication、RepNotify 与 RPC

> 例子：Replication 像共享表格，服务器把“血量现在是 80”同步给客户端；RepNotify 是表格变化后提醒界面和特效更新；RPC 像打电话通知“玩家刚刚按了开火”。

> 一句话：属性复制同步“结果是什么”，RPC 通知“发生了一件事”。

#### 3.14 网络预测、Reconciliation 与 Lag Compensation

> 例子：移动时客户端先按自己的账走，服务器账本回来后再对账，这叫预测和校正；射击时服务器会翻到玩家按下扳机那一刻的历史位置再判定，这叫延迟补偿。

> 一句话：预测让操作不卡，校正保证最终一致，延迟补偿让高延迟玩家的命中判断更公平。

#### 3.15 Dedicated Server（DS）与 Listen Server

> 例子：Listen Server 像房主一边打牌一边当裁判，房主退出房间也可能散；DS 像单独请了一个裁判服务器，自己不玩，只负责算输赢和同步所有玩家。

> 一句话：DS 是独立的权威服务器，常见于 FPS、搜打撤、MMO 和多人动作项目。

#### 3.4 UI 与资源

#### 3.16 UMG、Slate 与 MVVM

> 例子：UMG 像在编辑器里拖控件搭游戏界面；Slate 像更底层的 UI 零件库，做编辑器工具时常见；MVVM 像把“页面长什么样”和“数据从哪里来”分开，血量变了，界面自己跟着更新。

> 一句话：UMG 做游戏 UI，Slate 做底层和工具 UI，MVVM 负责把数据和界面拆开。

#### 3.17 Cook、Pak 与 IoStore

> 例子：美术资源和蓝图像厨房里的原材料，不能原样发给玩家。Cook 先把材料加工成目标平台能吃的格式，Pak/IoStore 再把成千上万个小文件装进几个大箱子，方便安装和读取。

> 一句话：Cook 是加工资源，Pak/IoStore 是把成品资源打包进运行时容器。

## 第四章 Unity 客户端

> GameObject、Mono/IL2CPP、热更新、Addressables、UI、动画、渲染与 DOTS。

#### 4.1 对象、运行时与热更新

#### 4.1 GameObject、Component 与 MonoBehaviour

> 例子：GameObject 像一块空插线板，本身只是一个对象；移动、碰撞、音频和脚本都靠 Component 插上去。MonoBehaviour 是最常见的脚本组件，引擎会在 Awake、Start、Update 等时间点自动来调用它。

> 一句话：Unity 用 GameObject 装对象，用 Component 拼功能，用 MonoBehaviour 写游戏逻辑。

#### 4.2 Mono 与 IL2CPP

> 例子：Mono 像带着翻译员边走边解释 C# 代码，开发调试方便；IL2CPP 像出发前先把 C# 翻译成 C++ 再编译成机器代码，更适合正式包和平台要求，但构建更慢、排查也更复杂。

> 一句话：Mono 偏开发和灵活，IL2CPP 偏正式发布和平台兼容。

#### 4.3 Unity GC 与每帧分配

> 例子：每帧只扔一个小纸团看起来不多，但一秒 60 帧、跑十分钟，垃圾桶很快就满。GC 一来集中清理，画面可能突然卡一下，所以 Unity 简历常写“减少 GC Alloc、对象池、无分配”。

> 一句话：Unity GC 优化的重点是少在每帧不停创建临时对象。

#### 4.4 xLua、HybridCLR 与代码热更新

> 例子：xLua 是让 C# 和 Lua 互相说话的翻译；HybridCLR 则让正式包里还能运行后续下发的新 C# 程序集。两者目的都像给游戏留一扇“换说明书”的门，活动和逻辑改动不用每次重新上架整包。

> 一句话：xLua 用 Lua 热更，HybridCLR 用 C# 热更。

#### 4.2 资源、界面与性能

#### 4.5 AssetBundle 与 Addressables

> 例子：AssetBundle 像把角色、皮肤和关卡分别装进不同快递箱；Addressables 像快递平台，只要按“角色/皮肤 A”这个地址下单，它会帮你找到在哪个箱子、依赖哪些箱子以及什么时候卸货。

> 一句话：AssetBundle 是资源包，Addressables 是管理资源地址、依赖、下载和释放的一套上层系统。

#### 4.6 Animator、Controller 与 Blend Tree

> 例子：AnimationClip 是一段走路或攻击录像，Controller 像节目单，规定什么时候播哪段；Animator 是播放器；Blend Tree 会按速度把走、慢跑和冲刺自然混在一起。

> 一句话：这组词表示候选人做过 Unity 角色动画状态和混合。

#### 4.7 UGUI 与 UI Toolkit

> 例子：UGUI 像在画布上拖按钮、图片和文本，是游戏运行时界面的主力；UI Toolkit 更像网页，用结构和样式表组织界面，编辑器工具和新项目里越来越常见。

> 一句话：UGUI 偏传统游戏 UI，UI Toolkit 偏结构化界面和编辑器工具。

#### 4.8 URP、HDRP 与 SRP

> 例子：URP 像覆盖手机、PC 的通用生产线，性能和效果比较平衡；HDRP 像设备更重的高端电影棚，画质更高但要求也更高；SRP 是允许团队自己改造整条渲染生产线的框架。

> 一句话：URP 偏通用和移动端，HDRP 偏高端画质，SRP 是可定制渲染管线。

#### 4.9 Job System、Burst 与 DOTS/ECS

> 例子：如果有十万个小兵，每个小兵都带一大本对象档案会很重。ECS 把所有位置放一排、所有血量放一排；Job System 把批量计算拆给多个工人；Burst 再把说明书编译得更快。

> 一句话：DOTS/ECS 是面向大量对象的批处理架构，Job 和 Burst 负责把计算跑得更快。

## 第五章 网络协议与服务端框架

> Socket、TCP/UDP、KCP、RPC、Protobuf、Actor、Erlang、Skynet 与 ET。

#### 5.1 通信协议

#### 5.1 Socket

> 例子：Socket 像墙上的网口或电话插座，它只是程序连接网络的入口。TCP、UDP 是不同的运输规则，真正的登录、移动和战斗消息才是插座里传的内容。

> 一句话：Socket 是程序收发网络数据的接口，不等于某一种具体协议。

#### 5.2 TCP

> 例子：TCP 像按编号寄出的挂号信，保证送到、保证顺序。第 3 封没到时，第 4、5 封通常也得先等着，所以可靠但遇到丢包时可能卡住。

> 一句话：TCP 可靠有序，适合登录、聊天、支付等不能随便丢的数据。

#### 5.3 UDP 与 KCP

> 例子：UDP 像直接扔明信片，快，但可能丢、重复或乱序。KCP 像在明信片通道上自己加签收和补发规则，愿意多发一点数据，换更低的等待时间。

> 一句话：UDP 追求低延迟，KCP 是在 UDP 上补可靠性的常见方案。

#### 5.4 HTTP 与 WebSocket

> 例子：HTTP 像去窗口办事，一次申请换一次答复，适合登录、公告和商城；WebSocket 像接通一条长期电话线，双方随时都能主动说话，适合聊天、大厅和实时通知。

> 一句话：HTTP 是一问一答，WebSocket 是长期双向连接。

#### 5.5 RPC、gRPC 与 Protobuf

> 例子：RPC 像你直接喊“库存服务，帮我扣一件货”，写起来像调用本地函数，其实中间要过网络。gRPC 是一套成熟的跨服务通话方案；Protobuf 是双方提前约好的紧凑表格格式。

> 一句话：RPC 让远程服务像函数一样调用，Protobuf 负责定义和压缩消息格式。

#### 5.2 并发模型与框架

#### 5.6 线程、协程与 Actor

> 例子：线程像真的多请几个员工；协程像一个员工把等外卖的单子夹书签，先处理别的；Actor 则像每个房间都有自己的管理员，别人不能直接改它的账，只能给它发消息。

> 一句话：线程是真并行单位，协程是轻量任务，Actor 用消息隔离共享数据。

#### 5.7 Erlang/OTP

> 例子：把服务器拆成许多互相发消息的小房间，每个房间自己管数据。某个房间的人出问题了，上级值班员可以按规则把它重启，而不是让整栋楼一起停工。

> 一句话：Erlang/OTP 擅长高并发、消息驱动和故障自动恢复。

#### 5.8 Skynet 与 ET

> 例子：Skynet 像用很多小办公室组成服务器，办公室之间发消息，业务常用 Lua 写；ET 则让 Unity 客户端和 C# 服务端使用相近的组件和代码习惯。简历写框架名不代表一定做过底层，仍要看负责了哪个模块。

> 一句话：Skynet 是 Lua/C 的游戏服务端框架，ET 是 C# 游戏全栈框架。

## 第六章 游戏服务器架构与基础设施

> 网关、匹配、战斗服、数据库、Redis、消息队列、Docker 与 K8s。

#### 6.1 服务器分工

#### 6.1 网关服

> 例子：大型园区不会让访客直接冲进财务室，先经过门卫和总机。网关服负责验身份、保持连接、限流，并把登录、聊天和战斗消息转给正确的后端服务。

> 一句话：网关服就是客户端进入服务器集群的统一门口和转发站。

#### 6.2 匹配服、房间服与战斗服

> 例子：匹配服像拼桌员，按照段位、网络和等待时间把人凑齐；房间服像牌桌管理员，管准备、选人和开局流程；战斗服像真正的裁判，算移动、技能和输赢。

> 一句话：匹配负责组人，房间负责局前流程，战斗服负责局内权威逻辑。

#### 6.3 DS 调度

> 例子：匹配成功不代表比赛已经有场地。调度系统要先找一台空服务器，或者临时启动一台 DS，再把地址发给玩家；比赛结束后还要回收机器。

> 一句话：DS 调度负责创建、分配、监控和回收每一局专用服务器。

#### 6.4 世界服、场景服与分区

> 例子：一座超大城市不可能让一个办公室管完，会按城区分给不同团队。玩家从上海区走进杭州区时，角色资料、周围玩家和任务状态也要安全交接。

> 一句话：世界服管全局关系，场景服管某张地图或某块区域的实时逻辑。

#### 6.2 数据与中间件

#### 6.5 MySQL 与事务

> 例子：MySQL 像字段严格的档案馆，账号、订单、角色资产都按表保存。充值发货时，扣款、写订单和加钻石要么一起成功，要么一起撤销，这种“几步一起算一笔账”就是事务。

> 一句话：MySQL 负责长期准确保存数据，事务保证一组关键操作不做一半。

#### 6.6 Redis

> 例子：Redis 像放在手边的高速临时柜，登录会话、排行榜、限流计数和热点数据都能快速取。它很快，但不能把所有长期资产只放这里，还要考虑过期、持久化和主从切换。

> 一句话：Redis 是高速缓存和共享状态工具，游戏服务端简历里非常常见。

#### 6.7 Kafka 与 RabbitMQ

> 例子：Kafka 像一条可以反复回看的流水传送带，适合日志、埋点和大量事件流；RabbitMQ 像快递分拣中心，把消息按规则投到不同信箱，处理完成后回执。

> 一句话：Kafka 偏大吞吐事件流，RabbitMQ 偏灵活路由和任务消息。

#### 6.8 etcd、Consul 与服务发现

> 例子：服务器每天会扩容、重启和换地址，不能靠手写通讯录。Consul 像实时公司电话簿，告诉你哪些服务还健康；etcd 像大家共同确认的关键公告板，K8s 也用它保存集群的重要状态。

> 一句话：它们负责保存关键配置、服务地址和集群状态。

#### 6.9 Docker 与 K8s

> 例子：Docker 像把程序和依赖装进标准集装箱，到哪台机器都能按同样方式打开；K8s 像港口调度中心，你只说“始终保持十个服务在岗”，它会自动安排机器、补人、重启和对外提供地址。

> 一句话：Docker 负责标准化打包，K8s 负责大规模部署、调度和自动恢复。

#### 6.10 幂等、分布式锁与数据一致性

> 例子：网络卡顿时，同一个“领取奖励”请求可能重发两次。幂等就是同一个订单号无论来几次都只发一份；分布式锁像多个窗口共用一把钥匙，同一时刻只允许一个窗口改这份关键数据。

> 一句话：幂等防重复，分布式锁防多个服务同时把同一份数据改乱。

#### 6.11 分片、复制与热点

> 例子：一个仓库放不下全部玩家，就按玩家 ID 分到不同仓库，这是分片；同一份关键档案多备几份，这是复制。但如果所有人同时抢王城数据，那一小块仍会排长队，这就是热点。

> 一句话：分片解决总量，复制解决可用性，热点是流量集中到少数数据。★ 6.12 Cs架构

CS架构是 客户端-服务器架构（Client-Server Architecture）的简称。它是一种软件架构模式，把系统分成两个主要部分：客户端（负责与用户交互）和服务器（负责提供服务和资源）。

结合你之前感兴趣的游戏本地化话题，绝大多数需要联网的游戏，都采用CS架构。我们可以用一个很直观的例子来理解：

想象你去一家高级餐厅：

你（顾客） = 客户端：你手里有菜单（用户界面），可以点菜（发出请求），可以吃饭（部分处理）。但你不能自己去厨房炒菜，也不能去酒窖拿酒。

餐厅的后厨、仓库、收银台 = 服务器：你点完菜，服务员（网络）把单子传给后厨。后厨负责备菜、烹饪、装盘（核心数据处理和存储）。你永远看不到后厨的运作细节。

服务员和传菜系统 = 网络：负责把你点的菜（请求）传给后厨，再把做好的菜（响应）端给你。

**CS架构的核心特点：**

分工明确：客户端只管“展示”和“交互”，服务器只管“计算”和“存储”。

一对多服务：一个服务器可以同时服务成千上万个客户端（比如《英雄联盟》的服务器服务所有玩家）。

客户端“瘦”或“胖”：

瘦客户端：大部分工作都在服务器做，本地只负责显示。就像你用浏览器玩网页游戏，电脑只是显示画面，逻辑都在云端。

胖客户端：部分工作也在本地做。比如你下载安装的《原神》，本地要负责渲染画面、处理物理碰撞等，服务器主要负责同步世界状态、验证数据、存储账号信息。

**在游戏中CS架构的作用：**

防作弊：关键的胜负判定（如你击中敌人的那一刻）在服务器进行，玩家无法篡改。

数据同步：所有玩家的位置、血量、动作等数据，都发送到服务器，再由服务器分发给其他玩家，让大家看到一致的世界。

持久化存储：你的等级、皮肤、充值记录都存在服务器上，换台电脑登录依然在。

你可能会遇到的另一种架构：BS架构

BS架构（Browser-Server，浏览器-服务器）是CS架构的一种特殊形式。它的客户端统一就是浏览器，不需要单独安装软件。比如你用Chrome浏览器玩一个H5网页小游戏，那就是BS架构。

**一句话总结：**

CS架构就是“我家的电脑/手机（客户端）和你家的服务器（服务器）互相配合，完成同一个任务”。 你每次联网登录《王者荣耀》或者《魔兽世界》，就是在使用一个典型的CS架构。

如果你想知道CS架构相比其他架构（比如区块链的“去中心化”架构）的优缺点，我可以进一步为你解释。

## 第七章 游戏网络同步

> 状态同步、帧同步、预测、校正、延迟补偿、Rollback、AOI 与弱网。

#### 7.1 同步模型

#### 7.1 状态同步

> 例子：服务器像唯一记账的裁判，定期告诉大家“角色现在在这里、血量是 80”。客户端在两次通知之间自己把动作补顺。FPS、MMO 和搜打撤里经常采用这种服务端权威思路。

> 一句话：状态同步传的是“现在世界是什么样”，服务器通常掌握最终结果。

#### 7.2 帧同步（Lockstep）

> 例子：几个人下同一盘棋，不用每次互发整张棋盘，只要广播“这一回合我走哪一步”。每个人在本地按同一套规则重算，就能得到同样棋局。MOBA、格斗和部分 RTS 常见。

> 一句话：帧同步主要同步玩家输入，要求每台机器算出完全一致的结果。

#### 7.3 固定 Tick 与确定性

> 例子：裁判规定每 20 毫秒正式记一次账，所有人都按同一个节拍计算；随机数也必须用同一颗种子，小数规则也要一致，否则几分钟后结果会越差越远。

> 一句话：固定 Tick 统一时间步，确定性保证相同输入一定得到相同结果。

#### 7.4 快照、插值与外推

> 例子：服务器像每隔一段时间拍一张角色位置照片。插值是在两张已经收到的照片之间平滑移动；外推是在下一张照片没来前，按刚才速度猜一小段；快照则是保存某个时刻的完整状态。

> 一句话：快照记录状态，插值负责平滑，外推负责短时间猜测未来。

#### 7.2 预测、回滚与兴趣管理

#### 7.5 客户端预测与 Reconciliation

> 例子：按下前进键后，客户端先让角色动，不等服务器；服务器回信说“你实际只走到这里”时，客户端把位置改成裁判版本，再重放之后的输入。

> 一句话：预测先给反馈，Reconciliation 再按服务器结果对账重算。

#### 7.6 Lag Compensation（延迟补偿）

> 例子：高延迟玩家开枪时，目标现在已经跑远了。服务器会像翻监控录像一样，回到玩家按下扳机的那一刻，看当时准星是否真的对准目标。

> 一句话：延迟补偿就是服务器回看历史状态来判断当时是否命中。

#### 7.7 Rollback

> 例子：双方先按猜到的输入继续打，真正输入到达后如果猜错，就退回几帧，把正确输入塞进去，再快速重演到现在。玩家通常只看到轻微修正。

> 一句话：Rollback 是“先玩、错了回去重算”，格斗和低延迟对战里常见。

#### 7.8 AOI

> 例子：一个 MMO 城里有十万人，但你没必要收到全城每个人的位置。AOI 像只订阅附近街区和与你有关的人，离得远、看不见的对象就不发。

> 一句话：AOI 只同步玩家关心的区域，降低服务器计算和网络带宽。

#### 7.9 服务器权威

> 例子：客户端可以说“我按了开火”，但不能直接说“我已经打死他并获得一万金币”。关键结果必须由服务器检查位置、冷却、伤害和资产后决定。

> 一句话：服务器权威就是客户端只报操作，最终结果由服务器说了算。

#### 7.10 Tick Rate、Send Rate 与带宽预算

> 例子：裁判每秒算 60 次，不代表每秒把全场所有数据复印 60 份。服务器会按重要程度决定谁多久发一次：正在向你开枪的人优先，远处不动的装饰物晚点发。

> 一句话：Tick Rate 是服务器计算频率，Send Rate 是发包频率，带宽预算决定先同步什么。

#### 7.11 弱网、重连与状态恢复

> 例子：网络像地铁进隧道，可能突然丢包、延迟暴涨或断开。重新连上时不能只把角色放回地图，还要补回当前局势、背包、血量和剩余时间，避免重复结算。

> 一句话：弱网处理包括丢包、抖动、超时、重连和重新对齐完整状态。

## 第八章 渲染与图形程序

> Shader、PBR/NPR、DrawCall、LOD、Nanite、Lumen、超分与 GPU 分析。

#### 8.1 画面怎样被算出来

#### 8.1 CPU、GPU 与帧预算

> 例子：CPU 像导演和场务，安排这一帧要画哪些人、发多少绘制命令；GPU 像摄影棚，真正把三角形和像素算出来。60 帧游戏每帧只有约 16.7 毫秒，哪边超时整帧都得等。

> 一句话：CPU 管调度和逻辑，GPU 管大量绘制计算，帧预算决定能不能稳定帧率。

#### 8.2 DrawCall、Batching 与 GPU Instancing

> 例子：导演让摄影棚画一百棵树，如果一棵一棵下命令，会花很多时间沟通。Batching 是把能合的东西分批一起画；Instancing 是告诉 GPU“同一种树按这一百个位置画一百份”。

> 一句话：DrawCall 是 CPU 给 GPU 的绘制命令次数，合批和实例化用来减少命令开销。

#### 8.3 Shader、HLSL 与 GLSL

> 例子：普通 C++ 代码主要让 CPU 做事，Shader 像给 GPU 的工序说明，让成千上万个小工人同时处理顶点、像素和光照。HLSL、GLSL 是不同图形平台常见的 Shader 语言。

> 一句话：Shader 决定模型和像素怎么被画出来，是渲染程序和 TA 的核心词。

#### 8.4 PBR 与 NPR

> 例子：PBR 像按真实材料规律做灯光：金属、塑料和布料在不同环境里仍然像对应材质；NPR 则像制定动画片画法，用描边、分段阴影和特殊高光故意做得“不真实但好看”。

> 一句话：PBR 偏写实材质，NPR 偏卡通和二次元风格。

#### 8.5 前向渲染与延迟渲染

> 例子：前向渲染像每上一道菜就当场把灯光和价格一起算完；延迟渲染像先把每道菜的材料信息记下来，最后统一算灯光。前向更直接，延迟更适合很多动态灯。

> 一句话：前向和延迟是两种主要渲染组织方式，各自在灯光数量、透明物体和性能上有取舍。

#### 8.2 优化与 UE5 技术

#### 8.6 LOD、HLOD 与剔除

> 例子：近处的楼要看清窗框，远处只需要一个简化外形；一整片远景建筑还能合成一个 HLOD。镜头完全看不到的东西就直接不画，这叫剔除。

> 一句话：LOD/HLOD 用距离降低模型成本，剔除负责不画看不见的对象。

#### 8.7 Nanite 与 Lumen

> 例子：Nanite 像自动帮高精度模型挑出镜头真正需要看的细节，不必美术手工做很多级 LOD；Lumen 像实时算房间里的反光和间接光，门打开、太阳移动后光线也会跟着变。

> 一句话：Nanite 解决高精度几何，Lumen 解决动态全局光照。

#### 8.8 TSR、DLSS 与 FSR

> 例子：游戏先画一张较小的图，再借助当前帧、历史帧和运动信息补成更清晰的大图，像把低清录像智能放大。DLSS 使用 NVIDIA 技术，FSR 来自 AMD，TSR 是 UE 的时序超分方案。

> 一句话：这些技术用较低渲染分辨率换更高帧率，再把画面补清楚。

#### 8.9 Overdraw、Shader Variant 与 PSO

> 例子：透明特效叠十层时，同一个像素可能被反复画十次，这叫 Overdraw；材质开关组合太多会产生大量 Shader Variant；第一次遇到某套显卡状态临时准备 PSO，还可能突然卡一下。

> 一句话：这三个词都和“画面看着不复杂却为什么卡”有关。

#### 8.10 RHI、RDG 与 RenderDoc/PIX/Nsight

> 例子：RHI 像翻译，把引擎的绘制命令翻成 DX12、Vulkan 或 Metal；RDG 像一张本帧渲染施工计划，安排工序和临时资源；RenderDoc、PIX、Nsight 则像逐步骤回放一帧，查到底哪次绘制最贵。

> 一句话：RHI/RDG 属于引擎渲染底层，抓帧工具用来定位 GPU 问题。

## 第九章 游戏 AI

> 状态机、行为树、黑板、导航、EQS、GOAP 与 AI Agent。

#### 9.1 规则驱动 AI

#### 9.1 FSM（有限状态机）

> 例子：守卫同一时间只处于巡逻、追击、攻击或逃跑中的一个状态。看到玩家就从巡逻切到追击，距离够近再切攻击。状态多了以后，连线会越来越复杂。

> 一句话：FSM 是“一个时刻一个状态，满足条件就切换”。

#### 9.2 行为树与 Blackboard

> 例子：行为树像 NPC 的办事流程：能攻击就攻击，不能就追，找不到人就巡逻；Blackboard 像随身小本子，记录目标是谁、上次在哪、血量多少，树里的各个节点都能查看。

> 一句话：行为树负责做决定，黑板负责保存当前上下文。

#### 9.3 NavMesh 与 A*

> 例子：NavMesh 先把地图上能走的地面铺成一张导航网；A* 像地图导航，一边看已经走了多远，一边估计离终点还有多远，优先找更有希望的路线。

> 一句话：NavMesh 表示哪里能走，A* 负责在可走区域里找路线。

#### 9.4 EQS、GOAP 与 Utility AI

> 例子：EQS 像先找出十个掩体再逐个打分；GOAP 像给 NPC 一个目标和行动规则，让它自己拼出步骤；Utility AI 则给攻击、躲避、治疗分别打分，选现在最值得做的。

> 一句话：EQS 选位置，GOAP 规划步骤，Utility AI 按分数选行为。

#### 9.5 AI Agent 与 LLM NPC

> 例子：大模型 NPC 像会聊天的店员，但它不能想干什么就干什么。设定资料由 RAG 提供，记忆保存过去对话，工具调用只开放“查询任务、播放动作、发放已批准奖励”等有限能力。

> 一句话：AI Agent 是让模型在规则和工具范围内理解、规划并执行任务。

## 第十章 工程化、构建与发布

> Git/P4、Jenkins、CI/CD、Cook、Pak、热更新、Crash 与性能回归。

#### 10.1 版本与自动化

#### 10.1 Git 与 Perforce（P4）

> 例子：Git 像每个人都带着一份代码时间线，适合分支和合并；P4 像所有人围绕一个中央大仓库工作，大型贴图和场景文件还能先锁住，避免两个人同时改坏。

> 一句话：Git 常见于代码协作，P4 在大型游戏和美术资源管理里很常见。

#### 10.2 Jenkins、CI 与 CD

> 例子：Jenkins 像一个不下班的构建同事。有人提交代码后，它自动拉取、编译、跑测试并发结果；CI 是持续检查代码能不能合，CD 是通过后继续自动做包和发布。

> 一句话：CI/CD 把编译、测试、打包和发布变成自动流水线。

#### 10.3 Cook、Pak 与热更新

> 例子：原始资源先 Cook 成手机、PC 或主机能高效读取的成品，再装进 Pak/IoStore。上线后若只改了一部分资源或脚本，就做补丁或热更新，不让玩家重新下载整包。

> 一句话：Cook 是加工，Pak 是装箱，热更新是只替换变化部分。

#### 10.4 Crash、符号、日志与埋点

> 例子：程序崩溃只留下一个地址时，符号文件像字典，能翻译成哪段代码哪一行；日志记录过程；埋点和指标记录玩家当时的帧率、网络、设备和业务状态。

> 一句话：它们共同把“玩家说崩了、卡了”变成能定位的证据。

#### 10.5 Profiler 与性能回归

> 例子：优化不是凭感觉删代码，而是先固定一段场景，用 Profiler 看 CPU、GPU、内存和加载时间花在哪。改完后用同样机器再跑一遍，若新版本又变慢，就叫性能回归。

> 一句话：Profiler 找瓶颈，性能回归测试防止优化成果被后续版本慢慢吃掉。

## 第十一章 技术美术（TA）

> 渲染 TA、管线 TA、角色 TA、动画 TA、特效 TA 与资产预算。

#### 11.1 TA 常见方向

#### 11.1 TA、渲染 TA 与管线 TA

> 例子：美术说“我要这种效果”，程序说“这么做会卡”，TA 像两边都听得懂的翻译。渲染 TA 主要做 Shader、材质和画面方案；管线 TA 主要用 Python、Maya、Houdini 做自动化工具和资源流程。

> 一句话：TA 是连接美术和程序的岗位，渲染 TA 管效果，管线 TA 管生产流程。

#### 11.2 Houdini 与程序化生成

> 例子：不是让美术一栋栋摆房子，而是先写“道路怎么分叉、楼房多高、窗户怎么排列”的规则，然后改几个参数就能生成很多街区版本。

> 一句话：Houdini 常用来按规则批量生成地形、道路、建筑和特效资源。

#### 11.3 Rig、Skin 与 Control Rig

> 例子：Rig 像给角色搭木偶骨架和控制器；Skin 决定皮肤怎么跟着骨头变形；Control Rig 则把一套可调的动画控制工具放进 UE，方便运行时或编辑器里修动作。

> 一句话：这组词通常对应角色 TA、绑定或动画技术岗位。

#### 11.4 Niagara、VFX 与资产预算

> 例子：Niagara 像一套粒子搭建器，用规则组合火、烟、爆炸和技能轨迹。TA 不只要让效果好看，还会规定粒子数、贴图大小、透明层数等预算，避免一场团战把机器压垮。

> 一句话：Niagara/VFX 管特效表现，资产预算管效果能不能稳定跑。

## 第十二章 引擎开发、测试与性能

> 任务系统、资源加载、编辑器工具、自动化测试、压测与性能分析。

#### 12.1 引擎与工具

#### 12.1 引擎开发与玩法客户端

> 例子：玩法客户端像在商场里开店，做技能、任务、UI 和玩法；引擎开发像修商场的电梯、供电、仓库和施工工具，处理内存、线程、渲染、资源、编辑器和跨平台底层。

> 一句话：玩法程序做游戏功能，引擎程序做支撑所有功能的底层系统和工具。

#### 12.2 TaskGraph、Job System 与任务调度

> 例子：不是每个系统都自己长期雇一个线程，而是把工作切成许多小任务，交给统一调度中心按依赖和空闲工人分配。这样能更充分利用多核 CPU。

> 一句话：任务系统把大工作拆小并统一调度，是引擎多线程的常见方式。

#### 12.3 异步加载、Streaming 与资源管理

> 例子：玩家往前走时，游戏在后台提前把前方地图和模型搬进来，同时把身后的内容卸掉。若一次性在主线程搬完，就会像搬家卡住整条路，出现明显掉帧。

> 一句话：异步加载和 Streaming 让资源边玩边进出内存，减少加载卡顿。

#### 12.4 Slate、编辑器扩展与工具开发

> 例子：游戏 UI 像给玩家用的前台，编辑器工具像给开发团队用的后台。工具开发会做批量检查、关卡编辑器、资源导入和一键构建，让几十个人少重复点几千次按钮。

> 一句话：编辑器和工具开发的价值是提升整个团队的生产效率。

#### 12.2 测试与性能

#### 12.5 自动化测试、接口测试与 Gauntlet

> 例子：让机器自己启动客户端、服务器，登录、跑图、战斗并核对结果，就是自动化测试；接口测试不打开游戏，直接发请求检查服务端；Gauntlet 能把 UE 客户端、DS 和设备一起拉起来跑完整流程。

> 一句话：测试开发不是只点界面，而是写程序让测试重复、批量、自动运行。

#### 12.6 负载、压力、Soak 与弱网测试

> 例子：正常一万人跑得住叫负载测试；继续加到系统撑不住叫压力测试；连续跑几天看内存会不会慢慢涨叫 Soak；再故意加延迟、丢包和断网，检查同步和重连。

> 一句话：这些测试用来确认系统上限、长期稳定性和差网络下的表现。

#### 12.7 CPU、GPU、内存与 I/O Profiling

> 例子：游戏卡顿像餐厅出餐慢，先要确认是厨师算得慢、烤箱太忙、仓库没空间，还是送菜堵路。CPU、GPU、内存和 I/O 工具分别看不同环节，不能只盯着平均帧率。

> 一句话：性能分析先分清瓶颈在哪一类，再找具体函数、资源或渲染 Pass。

## 第十三章 项目专项与岗位速查

> MMO、SLG、FPS、搜打撤、3C、反作弊与常见岗位关键词。

#### 13.1 项目里常见的组合词

#### 13.1 MMO：AOI、分服、场景服与跨服

> 例子：MMO 像一座长期营业的大城市：玩家被分到不同世界和地图服务，附近的人通过 AOI 同步；副本是单独开的房间；跨服玩法则把不同服务器的人临时送到一个中立赛场。

> 一句话：看到这些词，通常是大型在线世界、服务端架构或大地图客户端经验。

#### 13.2 SLG：大地图、时间轮、寻路与战报

> 例子：SLG 像一张一直运行的棋盘。部队不用每秒真的挪动，只记录出发时间、路线和到达时间；时间轮像未来闹钟盒子；战斗结果常由服务器先算完，再给客户端播放战报动画。

> 一句话：SLG 高频词集中在大地图、定时事件、联盟、战报和高并发服务端。

#### 13.3 FPS/TPS：Hitbox、延迟补偿、Rollback 与 DS

> 例子：玩家看到准星打中，不代表服务器现在看到的目标还在那里。Hitbox 是命中盒，延迟补偿会回看历史位置，部分系统还会回滚重演；DS 则是独立裁判，防止房主作弊。

> 一句话：这些词通常对应射击客户端、网络同步、战斗服务端或反作弊。

#### 13.4 搜打撤：Loot、局内背包与撤离结算

> 例子：玩家进局捡装备像把临时货物装进背包，只有成功撤离后才能真正带回仓库。断线、死亡、组队掉落和重复结算都要严格处理，否则物品会凭空复制或丢失。

> 一句话：搜打撤除了射击网络，还特别看重物品状态和撤离结算。

#### 13.5 3C：Character、Camera、Control

> 例子：同一个角色手感不好，可能不是战斗技能的问题，而是角色移动、摄像机跟随和输入控制没有配好。3C 就是把这三件事一起调，让走路、转向、瞄准和镜头反馈形成完整手感。

> 一句话：3C 是角色、镜头和操作控制的整体体验，动作和开放世界项目常见。

#### 13.6 反作弊、回放与战斗录像

> 例子：反作弊像商场既装门禁，也看异常行为：客户端检测改内存，服务器检查不可能的移动和伤害；回放则把每步输入或状态保存下来，既能观战，也能复盘举报和线上问题。

> 一句话：反作弊防非法行为，回放提供证据、观战和问题复现。

## 第十四章 候选人技术实力判断

> 把前面的术语、方向、阶段和技术线合在一起，避免只凭关键词下结论。

#### 14.1 简历里出现哪些技术栈，通常是强能力信号

- 性能优化：CPU / GPU / 内存 / I/O / 包体 / 启动速度 / 加载 / 发热 / Profiler / RenderDoc / PIX / Nsight。
- 网络与同步：状态同步、帧同步、预测、Reconciliation、Rollback、Lag Compensation、AOI、带宽预算、弱网恢复、DS。
- 引擎与底层：UE/Unity 源码、UObject、TaskGraph、RHI、RDG、资源系统、异步加载、跨平台、自研引擎。
- 渲染与图形：Shader、PBR/NPR、渲染管线、DX12/Vulkan/Metal、Nanite/Lumen、PSO、GPU 分析。
- 服务端架构：高并发、分片、容灾、幂等、数据一致性、K8s、Redis、Kafka、etcd、网关、世界服、战斗服。
- 战斗 / 3C / 物理：完整战斗框架、技能系统、CMC、Root Motion、动画、相机、操作手感、物理和载具同步。
- 工具与工程化：编辑器工具、自动化构建、CI/CD、构建农场、热更新、Crash、性能回归。
- TA / 测试开发 / 安全：Shader 工具化、Houdini、资产预算、自动化平台、压测、Soak、弱网、反外挂。
#### 14.2 不要因为出现一个高价值名词，就直接判断候选人很强

| 简历写法 | 通常说明什么 | 应该继续问什么 |
| --- | --- | --- |
| “用过 / 接触过 / 参与过 GAS、K8s、DS” | 可能只是项目环境里有，个人深度未知。 | 具体负责哪个模块？哪些决策由你做？ |
| “负责 / 设计 / 重构 / 主导” | 更可能有模块所有权，但仍需验证。 | 原来有什么问题？为什么这么设计？ |
| “优化性能” | 如果无数据，可能只是常规改动。 | 用什么工具定位？改前改后是多少？ |
| “项目百万 DAU / 十万 CCU” | 是团队规模，不自动等于个人能力。 | 你负责的服务或场景承载多少？故障时怎么处理？ |
| “做了很多 UI / 系统功能” | 通常是业务执行经验多。 | 有没有框架、性能、复杂交互或跨平台难点？ |

#### 14.3 十分制快速判断表

| 维度 | 分值 | 怎么判断 |
| --- | --- | --- |
| 技术深度 | 0-3 | 是否涉及底层原理、复杂问题和方案取舍。 |
| 模块所有权 | 0-2 | 是参与、负责，还是从设计到线上长期主导。 |
| 结果证据 | 0-2 | 是否有帧率、内存、包体、延迟、容量、效率等量化结果。 |
| 项目阶段 | 0-1 | 是否经历过 0-1、量产、上线或线上事故。 |
| 规模与复杂度 | 0-1 | 是否面对真实大规模、多平台、多端或高并发场景。 |
| 迁移能力 | 0-1 | 能否说明底层原则，而不是只会某个项目里的按钮。 |

> 分数怎么读8-10 分：核心技术候选人，适合高要求或稀缺岗位。6-7 分：成熟骨干，模块能独立负责。4-5 分：稳定执行，匹配常规业务岗。0-3 分：信息不足或经历偏浅，需要继续核实。
