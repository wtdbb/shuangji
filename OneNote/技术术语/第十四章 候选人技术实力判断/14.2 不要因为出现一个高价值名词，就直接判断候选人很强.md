# 14.2 不要因为出现一个高价值名词，就直接判断候选人很强

| 简历写法 | 通常说明什么 | 应该继续问什么 |
| --- | --- | --- |
| “用过 / 接触过 / 参与过 GAS、K8s、DS” | 可能只是项目环境里有，个人深度未知。 | 具体负责哪个模块？哪些决策由你做？ |
| “负责 / 设计 / 重构 / 主导” | 更可能有模块所有权，但仍需验证。 | 原来有什么问题？为什么这么设计？ |
| “优化性能” | 如果无数据，可能只是常规改动。 | 用什么工具定位？改前改后是多少？ |
| “项目百万 DAU / 十万 CCU” | 是团队规模，不自动等于个人能力。 | 你负责的服务或场景承载多少？故障时怎么处理？ |
| “做了很多 UI / 系统功能” | 通常是业务执行经验多。 | 有没有框架、性能、复杂交互或跨平台难点？ |

---
### 💡 猎头应用与常见排雷（补充）
- **简历搜索与匹配**：如果候选人简历中频繁提及 **不要因为出现一个高价值名词，就直接判断候选人很强** 相关词汇，说明其参与了对应的技术模块或具备相应底层经验。
- **面试与沟通对齐**：不要仅停留在“是否了解该名词”，重点追问候选人在 **不要因为出现一个高价值名词，就直接判断候选人很强** 方面的**实际落地项目**、**遇到过的性能/技术瓶颈**以及**最终的量化成果（如优化的帧率、减少的耗时）**。
- **避坑与定级指南**：许多初中级人员仅停留在“调包”或“在框架内填代码”的阶段。若是推荐大厂高职级或核心引擎组岗，务必核实其是否具备底层机制拆解、架构重构和从0到1搭框架的能力。
